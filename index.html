10 / COMMENT C'EST CONSTRUIT

La stack technique.

Pas besoin de comprendre tout le détail technique. L'idée c'est que tu saches sur quoi ça repose et pourquoi ces choix.

Next.js 16

Le framework principal. C'est lui qui gère tout : les pages, les API, le rendu. Un seul projet pour le front et le back.

TypeScript

Comme du JavaScript, mais avec des types. Ça évite les bugs idiots et ça rend le code plus solide.

TailwindCSS v4

Pour le design. On écrit le style directement dans le HTML. C'est rapide et ça garde tout cohérent.

Supabase

La base de données (PostgreSQL) + l'authentification + le stockage des fichiers. Tout en un.

Claude AI (Anthropic)

L'IA qui analyse les avis et génère les réponses. Deux modèles : Haiku (rapide, pas cher) et Sonnet (plus qualité).

Apify

Service de scraping. Il n'y a pas d'API Google officielle pour les avis, donc on passe par Apify pour les récupérer.

Vercel

L'hébergement pour le moment. Déploiement automatique + cron jobs intégrés. On migrera sur le VPS ensuite.

SheetJS

Pour l'export Excel. Le courtier télécharge ses données en .xlsx pour la conformité ACPR. Un clic.

OAuth (MS365 + Gmail)

Pour les emails. L'idée c'est que les mails partent de la boite du courtier, pas d'un service tiers.

11 / SUPABASE

La base de données.

Supabase c'est une base PostgreSQL hébergée avec plein de trucs intégrés : authentification, stockage de fichiers, et surtout le RLS (Row-Level Security) qui fait que chaque courtier ne voit que ses propres données.

Les 8 tables :

courtiers

La table centrale. Nom, cabinet, SIRET, adresse, fiche Google, couleur, logo, style IA.

campagnes

Les campagnes de sollicitation. Nom, statut, compteurs (envoyés, ouverts, complétés).

sollicitations

Chaque client sollicité. Token unique, statut, relances, opt-out.

avis_internes

Les avis reçus via la page tampon. Note, texte, sentiment, thème, suggestion IA.

alertes

Les avis négatifs. Quand un client met 3 étoiles ou moins, une alerte est créée.

avis_google

Les avis Google scrapés. Auteur, note, texte, 3 suggestions de réponse IA.

concurrents

Les courtiers concurrents surveillés. Note, nb avis, évolution.

connections

Les tokens OAuth (Gmail/Microsoft). Chiffrés en AES-256.

12 / L'INTELLIGENCE ARTIFICIELLE

Claude fait le boulot d'analyse.

On utilise l'IA d'Anthropic (Claude). Deux modèles différents selon le besoin :

Claude Haiku

Le modèle rapide et pas cher. C'est lui qui analyse chaque avis interne : il détecte le sentiment (positif/neutre/négatif), le thème (service, tarif, délai...), les points de friction, et il génère une suggestion de réponse.

Avis arrive → Haiku analyse → JSON structure en retour

Claude Sonnet

Le modèle plus puissant. C'est lui qui génère les 3 suggestions de réponse pour les avis Google (professionnel, empathique, direct). La qualité doit être au top parce que ces réponses vont sur Google.

Avis Google → Sonnet → 3 réponses au choix

L'IA est personnalisée par courtier. Elle reçoit dans son prompt : le nom du cabinet, le style (formel/décontracté), la spécialité, et si le courtier a uploade son document ACPR, le contenu est aussi injecte. Du coup chaque courtier a des réponses différentes.

13 / APIFY — LE SCRAPING GOOGLE

Pas d'API Google pour les avis.

Google ne fournit pas d'API officielle pour récupérer les avis. Donc on utilise Apify, un service de scraping, pour aller lire les fiches Google des courtiers et de leurs concurrents.

Ce qu'Apify fait pour nous :

• Récupère les avis Google du courtier (nouveaux, modifiés, supprimés)
• Détecte les courtiers concurrents par zone géographique
• Suit l'évolution des notes des concurrents chaque semaine
• Tout ça tourne automatiquement avec des cron jobs (tâches planifiées)
Acteur Apify: compass~crawler-google-places — jusqu'à 40 avis par fiche

Le courtier crée une campagne, ajoute ses clients (un par un ou par CSV), personnalisé le message, et envoie. Chaque client reçoit un email avec un lien unique.

Il donne un nom

Exemple : "Clients janvier" ou "Après sinistres"

Il envoie

17 / LE ROUTAGE INTELLIGENT

Content → Google. Pas content → interne.

C'est la logique centrale. On ne redirigé jamais un client mécontent vers Google. Les bons avis vont sur Google, les mauvais restent en interne pour que le courtier puisse réagir.

★★★★+

4 ou 5 étoiles

Le client est content. On lui montre un bouton pour laisser un avis sur Google avec le lien direct vers la fiche du courtier.

★★★−

3 étoiles ou moins

L'avis reste en interne. Message de remerciement. Si le sentiment est négatif, une alerte est créée automatiquement pour le courtier.

26 / SÉCURITÉ

Les données sont protégées.

Pas un détail secondaire. Voici ce qui est en place :

AES-256-GCM

Les tokens OAuth (Gmail/Microsoft) sont chiffres avant d'aller en base. Jamais stockés en clair.

Row-Level Security

Chaque courtier ne voit que ses propres données. Même si quelqu'un essaie, la base bloque.

Middleware d'auth

Chaque requête est vérifiée. Si pas connecté, redirigé vers /login. Les pages publiques (tampon, webhooks) sont exclues.

CSP + HSTS

Headers de sécurité configurés. Pas d'iframe, HTTPS force, scripts limites.

31 / LA PARTIE LA PLUS COMPLEXE

Les emails depuis la boite du courtier.

C'est la partie qui m'a pris le plus de temps. L'idée c'est simple : quand le courtier envoie une campagne, les emails partent de sa propre adresse email (son Outlook ou son Gmail), pas d'un "noreply@hibroker.com". Le client reçoit un mail de "jean@cabinet-dupont.be" et il reconnaît l'expéditeur. Zero spam.

Comment ça marche :

1. Le courtier connecté sa boite mail pendant l'onboarding (un clic, OAuth)
2. On récupère un access_token + refresh_token
3. Les tokens sont chiffrés en AES-256-GCM avant d'aller en base
4. À chaque envoi : déchiffrement → appel API Microsoft Graph ou Gmail → email envoyé
5. L'email apparaît dans les "envoyés" du courtier
